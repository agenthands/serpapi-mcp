**1. `## Engine Loading Pipeline`** (per D-05: second required diagram, ARCH-03)

Both Mermaid (`flowchart TD`) and ASCII diagrams showing:
```
A[engines/ directory] --> B[os.ReadDir: list .json files]
B --> C{Filename matches ^[a-z0-9_]+.json?}
C -->|No| D[Skip with warning log]
C -->|Yes| E[Read & parse JSON]
E --> F{engine field == filename stem?}
F -->|No| G[Return error: fail-fast]
F -->|Yes| H[Add to schemas map + names list]
H --> I[Sort engine names alphabetically]
I --> J[Cache: engineNamesStore + schemasStore]
J --> K[Register serpapi://engines index resource]
J --> L[Register serpapi://engines/{name} per-engine resources]
K --> M[Return engineCount]
L --> M
```

Paragraph explaining:
- All ~107 engine schemas loaded at startup, fail-fast on any error
- Filename validation: `[a-z0-9_]+` pattern, non-matching files skipped with warning
- Strict validation: engine JSON field must match filename stem (e.g., `google.json` must have `"engine": "google"`)
- Resource handler closures capture pre-serialized JSON to avoid re-serialization per request
- Accessor functions (`EngineNames`, `RequiredParams`) read from cached stores

**2. `## Search Execution`** (per D-05: third required diagram, ARCH-04)

Both Mermaid and ASCII diagrams showing:
```
A[MCP search tool call] --> B[Unmarshal input: {params, mode}]
B --> C[Default mode: "complete"]
C --> D[Default engine: "google_light"]
D --> E[ValidateEngine: binary search in sorted list]
E -->|Invalid| F[toolError("invalid_engine")]
E -->|Valid| G[ValidateMode: "complete"|"compact"]
G -->|Invalid| H[toolError("invalid_mode")]
G -->|Valid| I[ValidateRequiredParams: check required:true params]
I -->|Missing| J[toolError("missing_params")]
I -->|OK| K[Extract API key from context]
K -->|Empty| L[toolError("missing_api_key")]
K -->|Present| M[Construct SerpApi URL with params + api_key]
M --> N[HTTP GET with 30s timeout]
N --> O{Status code?}
O -->|200| P[Parse JSON response]
O -->|429| Q[toolError("rate_limited")]
O -->|401| R[toolError("invalid_api_key")]
O -->|403| S[toolError("forbidden")]
O -->|Other| T[toolError("search_error", status + body)]
P --> U{Mode?}
U -->|compact| V[Remove metadata fields from resultMap]
U -->|complete| W[Return full resultMap]
V --> W
W --> X[Marshal to JSON + return CallToolResult]
```

Paragraph explaining:
- Input validation runs BEFORE any HTTP call (fail-fast)
- Three validation stages: engine name, mode, required params
- API key extracted from context (set by auth middleware)
- Shared `http.Client` with persistent connection pool, per-request 30s timeout via context
- All errors returned as MCP-compliant `IsError=true` with flat JSON body `{"error": code, "message": msg}`
- Compact mode removes top-level metadata keys for smaller responses

**3. `## Authentication Design`** (ARCH-06 partial — auth is a cross-cutting concern)

Paragraph explaining:
- Two auth methods: Bearer header (`Authorization: Bearer {KEY}`) takes priority, then path-based (`/{KEY}/mcp`)
- Path stripping: auth middleware removes the key segment before forwarding to mux
- Context propagation: API key stored via custom `contextKey` type, retrieved with `APIKeyFromContext()`
- /health exempt from authentication
- Auth-disabled mode: `--auth-disabled` flag or `MCP_AUTH_DISABLED` env var, uses `authOrPassthrough()` wrapper
- 401 response: `{"error": "Missing API key..."}` with detailed message about both methods

**4. `## Observability`** (ARCH-06)

Paragraph explaining:
- Correlation IDs: 32-char hex from crypto/rand, client can provide via `X-Correlation-ID` header or server generates
- Context propagation: `CorrelationIDFromContext()` used by search tool for structured logging
- Response header: correlation ID echoed back in `X-Correlation-ID` response header
- Structured logging: `slog` throughout, with `correlation_id`, `engine`, `mode`, `address`, `version` fields
- Request lifecycle: engine loading logs count at startup, search logs request params, server logs address+version on start

**5. `## Error Handling`** (cross-cutting)

Paragraph explaining:
- Auth errors: 401 JSON `{"error": "Missing API key..."}`
- Search tool errors: MCP `IsError=true` with flat JSON `{"error": "<code>", "message": "<desc>"}` (per D-01, D-02 in Phase 03 decisions)
- Error codes: `invalid_engine`, `invalid_mode`, `missing_params`, `missing_api_key`, `search_error`, `rate_limited`, `invalid_api_key`, `forbidden`
- SerpApi HTTP status mapping: 200→success, 429→rate_limited, 401→invalid_api_key, 403→forbidden, other→search_error with body
- All tool errors are non-fatal — returned as results, not Go errors, so MCP protocol handles them correctly

  2. `actions/setup-go@v5` — Go 1.25.x with module cache
  3. golangci-lint action (latest version)
  4. `go vet ./...`

**Release Flow** (.goreleaser.yml):
- Triggered by version tags (e.g., `v1.0.0`)
- Build: `CGO_ENABLED=0` static binaries
- Platforms: linux/amd64, linux/arm64, darwin/amd64, darwin/arm64, windows/amd64 (windows/arm64 excluded)
- Archives: tar.gz (zip for Windows)
- Checksums: SHA256 in checksums.txt
- Ldflags: `-s -w -X main.version={{.Version}} -X main.commit={{.Commit}} -X main.date={{.Date}}`
- Release mode: replaces (updates existing GitHub release)

**Makefile Targets** — table:

| Target | Command | Purpose |
|--------|---------|---------|
| `test` | `go test -count=1 ./...` | Run all tests |
| `test-race` | `go test -race -count=1 ./...` | Run tests with race detector |
| `cover` | `go test -coverprofile=coverage.out -covermode=atomic ./...` | Generate coverage report |
| `lint` | `golangci-lint run ./...` | Run linter |
