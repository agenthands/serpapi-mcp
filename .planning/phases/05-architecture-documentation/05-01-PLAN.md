### `cmd/serpapi-mcp` — Entry Point
- Purpose: CLI entry point, flag parsing, server initialization orchestration
- Key types: none exported (main package)
- Key functions:
  - `run(ctx, args, stdout, stderr) error` — all startup logic, extracted from main() for testability
  - `envOr(key, fallback) string` — env var with string fallback
  - `envIntOr(key, fallback) int` — env var with int fallback
  - `envBoolOr(key, fallback) bool` — env var with bool fallback (1/true/yes)
- Build-time variables: `version`, `commit`, `date` — injected via goreleaser ldflags
- Dependencies: `internal/server`, `internal/engines`, `internal/search`

### `internal/server` — MCP Server, Auth & CORS
- Purpose: HTTP server lifecycle, MCP protocol routing, authentication, CORS
- Key types:
  - `Config` — server configuration (Host, Port, CorsOrigins, AuthDisabled)
  - `MCPServer` — wraps mcp.Server with HTTP handler, healthcheck, graceful shutdown
  - `CORSConfig` — allowed origins list
- Key functions:
  - `NewMCPServer(cfg, version) *MCPServer` — creates MCP server with streamable HTTP transport
  - `buildHandler() http.Handler` — constructs chain: CORS → correlation → auth → mux
  - `Run(ctx) error` — starts serving with graceful shutdown (10s timeout)
  - `APIKeyFromContext(ctx) string` — extracts API key from request context
  - `NewCORSConfig(origins) *CORSConfig` — parses comma-separated origins string
- Auth middleware: Bearer header (priority) → path /{KEY}/mcp → 401 with JSON error
- Dependencies: `internal/middleware`, `github.com/modelcontextprotocol/go-sdk/mcp`

### `internal/engines` — Engine Schema Loading & Discovery
- Purpose: Load, validate, and serve SerpApi engine JSON schemas as MCP resources
- Key types:
  - `engineSchema` (unexported) — holds engine name + raw JSON
- Key functions:
  - `LoadAndRegister(srv, enginesDir, logger) (int, error)` — loads all .json files, validates, registers MCP resources
  - `EngineNames() []string` — returns sorted list of loaded engine names
  - `RequiredParams(engineName) []string` — extracts params with required:true from schema
  - `registerEnginesIndex(srv, names, logger)` — serpapi://engines index resource
  - `registerEngineResource(srv, name, schema, logger)` — serpapi://engines/{name} resource
- Validation: filename must match `[a-z0-9_]+.json`, engine field must match filename stem
- Fail-fast: returns error on first invalid/mismatched file
- Dependencies: `github.com/modelcontextprotocol/go-sdk/mcp`

### `internal/search` — Search Tool & Validation
- Purpose: MCP search tool implementation with input validation and SerpApi HTTP calls
- Key functions:
  - `RegisterSearchTool(srv, logger)` — registers "search" tool on MCP server
  - `callSearchTool(ctx, req) (*CallToolResult, error)` — core handler: unmarshal → validate → HTTP call → response
  - `ValidateEngine(engine) error` — binary search against loaded engine list
  - `ValidateMode(mode) error` — "complete" or "compact"
  - `ValidateRequiredParams(engine, params) error` — checks required:true params present
  - `toolError(code, message) *CallToolResult` — flat JSON `{"error": code, "message": msg}` with IsError=true
- Compact mode removes: search_metadata, search_parameters, search_information, pagination, serpapi_pagination
- Default engine: "google_light", default mode: "complete"
- 30-second per-request timeout via context
- Dependencies: `internal/engines`, `internal/middleware`, `internal/server`

### `internal/middleware` — HTTP Middleware
- Purpose: Cross-cutting HTTP middleware for request correlation
- Key functions:
  - `CorrelationIDMiddleware(next) http.Handler` — injects 32-char hex correlation ID
  - `CorrelationIDFromContext(ctx) string` — extracts correlation ID
  - `generateCorrelationID() string` — crypto/rand → 16 bytes → 32-char hex
- Constants:
  - `CorrelationIDHeader = "X-Correlation-ID"` — header name for client-provided or generated ID
- Dependencies: none (only stdlib)

**2. HTTP Request Flow section** (per D-05: the request flow diagram — ARCH-02)

Add a section titled `## HTTP Request Flow` with both Mermaid and ASCII diagrams showing the handler chain in `buildHandler()`:

Mermaid diagram (`flowchart TD`):
```
A[Incoming HTTP Request] --> B{OPTIONS?}
B -->|Yes| C[CORS Middleware: respond 204 No Content]
B -->|No| D[CORS Middleware: add headers, continue]
D --> E[Correlation ID Middleware: generate or accept X-Correlation-ID]
E --> F{Auth Disabled?}
F -->|Yes| H[Mux Router]
F -->|No| G{Has API Key?}
G -->|Bearer header or /KEY/mcp| H
G -->|No key| I[401 Unauthorized: JSON error]
H --> J{Path?}
J -->|/mcp| K[MCP Streamable HTTP Handler]
J -->|/health| L[Healthcheck: {"status":"healthy"}]
K --> M[SerpApi HTTP Call]
M --> N[Response: JSON or toolError]
```

ASCII diagram showing same flow with text-art arrows.

Include a paragraph explaining:
- Handler chain ordering: CORS first (so OPTIONS preflight gets CORS headers without hitting auth), then correlation ID, then auth, then mux router
- Auth priority: Bearer header first, then path-based `/{KEY}/mcp`
- /health is exempt from authentication
- Auth-disabled mode (`--auth-disabled` flag or `MCP_AUTH_DISABLED` env var) skips auth middleware entirely via `authOrPassthrough`
