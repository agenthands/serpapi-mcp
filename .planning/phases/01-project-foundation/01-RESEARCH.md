## Architecture Patterns

### Go Module Initialization
- Module path: `github.com/agenthands/serpapi-mcp` (per D-15)
- `go mod init` creates go.mod; `go get` adds go-sdk
- Import path for MCP: `github.com/modelcontextprotocol/go-sdk/mcp`

### Standard Go Project Layout
```
cmd/serpapi-mcp/main.go       # Entry point
internal/server/               # MCP server logic (Phase 2)
internal/search/               # Search tool logic (Phase 3)
internal/engines/               # Engine resource logic (Phase 2)
go.mod
go.sum
```
Per D-17: domain-based internal packages matching Python server's logical structure.

### go-sdk Server Creation Pattern (Phase 2 Reference)
```go
server := mcp.NewServer(&mcp.Implementation{
    Name:    "serpapi-mcp",
    Version: "1.0.0",
}, nil)
```

### go-sdk StreamableHTTPHandler (Phase 2 Reference)

handler := mcp.NewStreamableHTTPHandler(func(r *http.Request) *mcp.Server {

### Phase 1 main.go Only Needs

- Actual MCP server wiring is Phase 2

## Don't Hand-Roll

- Don't hand-roll MCP protocol — use go-sdk
- Don't hand-roll HTTP handler — use go-sdk's StreamableHTTPHandler (Phase 2)
- Don't write custom linters — golangci-lint defaults are sufficient

## Common Pitfalls

1. **go.mod vs go.sum**: Both must be committed; `go.sum` is auto-generated by `go get`
2. **CGO_ENABLED=0**: Required in goreleaser and CI for static binary cross-compilation
3. **golangci-lint version**: Pin in CI to avoid breaking changes (`golangci/golangci-lint-action@v7`)
4. **goreleaser snapshot mode**: `goreleaser release --snapshot --clean` for build verification without creating a real release
5. **Legacy archival order**: Move Python files BEFORE adding Go code to avoid confusing git history — but actually, since legacy move and Go init are in same plan, order within tasks matters

## Validation Architecture

### Observable Behaviors to Validate
1. `go build ./cmd/serpapi-mcp` produces a binary — automated via go build
2. `golangci-lint run && go vet ./...` passes — automated via tools
3. Legacy Python code is in `legacy/` and out of Go build path — automated via file existence check
4. `goreleaser release --snapshot` produces 5 platform binaries — automated via goreleaser

### Validation Approach
- All Phase 1 success criteria are automatable via CLI commands
- No manual UI or interaction needed
- Validation tasks built into plan verify steps
