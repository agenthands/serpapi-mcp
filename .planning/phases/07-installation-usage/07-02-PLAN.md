1. **Title and intro** — "Usage" heading, brief description: "How to configure, run, integrate, and troubleshoot the SerpApi MCP Server."

2. **Configuration and running** (USE-01)
   - **CLI flags reference table:**
     | Flag | Env var | Default | Description |
     |------|---------|---------|-------------|
     | --host | MCP_HOST | 0.0.0.0 | Host to bind to |
     | --port | MCP_PORT | 8000 | Port to bind to |

   - Starting the server: `serpapi-mcp` (uses defaults), `serpapi-mcp --port 3000`, `MCP_PORT=3000 serpapi-mcp`
   - Version check: `serpapi-mcp --version`
   - .env file support note: "The server reads environment variables directly (no .env file loading). Use shell env vars or a process manager."

3. **API key authentication** (USE-02, D-07/D-08)
   - **Path-based** (recommended, per D-08): `/{YOUR_API_KEY}/mcp`
     - Self-hosted: `http://localhost:8000/YOUR_API_KEY/mcp`
     - Hosted: `https://mcp.serpapi.com/YOUR_API_KEY/mcp`
   - **Header-based**: `Authorization: Bearer YOUR_API_KEY`
   - Auth-disabled mode for testing: `--auth-disabled` or `MCP_AUTH_DISABLED=true`
   - Priority: Bearer header takes precedence over path (per auth.go)
   - /health endpoint is exempt from authentication

4. **MCP client integration** (USE-03, D-07/D-08/D-09)
   - **Transport note** (D-09): The server uses Streamable HTTP protocol. No SSE or WebSocket alternatives.
   - **Claude Desktop** config:
     ```json
     {
       "mcpServers": {
         "serpapi": {
           "url": "https://mcp.serpapi.com/YOUR_API_KEY/mcp"
         }
       }
     }
     ```

   - **Cursor** config:

5. **Engine discovery and search** (USE-04)

   - **Per-engine resource**: `serpapi://engines/google_light` returns the full parameter schema
   - How to list available engines via MCP resource
   - How to get a specific engine's parameters
   - **Search tool usage**: `search` tool with `params` dict and optional `mode`

     - Specific engine: `{"params": {"q": "search query", "engine": "google"}}`
     - Compact mode: `{"params": {"q": "search query"}, "mode": "compact"}`
     - With location: `{"params": {"q": "coffee shops", "engine": "google", "location": "Austin, TX"}}`

6. **Error reference** (USE-05, D-10)
   - Complete error response catalog with exact JSON format from auth.go and search.go:
     - **401 Missing API key** (from auth.go line 87-88):
       ```json
       {"error": "Missing API key. Use path format /{API_KEY}/mcp or Authorization: Bearer {API_KEY} header"}
       ```
       Cause: No API key in path or header. Fix: Include API key in URL path or Authorization header.
     - **invalid_engine** (from validation.go):
       ```json
       {"error": "invalid_engine", "message": "invalid_engine: engine 'googl' not found. Available engines: baidu, bing, duckduckgo, google, google_light, ..."}
       ```
       Cause: Engine name typo or unsupported engine. Fix: Check `serpapi://engines` for valid names.
     - **invalid_mode** (from validation.go):
       ```json
       {"error": "invalid_mode", "message": "invalid_mode: mode must be 'complete' or 'compact', got 'fast'"}
       ```
       Cause: Mode value not "complete" or "compact". Fix: Use "complete" or "compact".
     - **missing_params** (from validation.go):
       ```json
       {"error": "missing_params", "message": "missing_params: missing required parameter(s) q for engine 'google'"}
       ```
       Cause: Required parameters omitted. Fix: Check `serpapi://engines/<engine>` for required params.
     - **missing_api_key** (from search.go):
       ```json
       {"error": "missing_api_key", "message": "No API key found in request context. Use path format /{API_KEY}/mcp or Authorization: Bearer {API_KEY} header."}
       ```
       Cause: Auth middleware bypassed (auth-disabled mode) but no key in context. Fix: Use auth-enabled mode or pass key via header.
     - **rate_limited** (from search.go, HTTP 429):
       ```json
       {"error": "rate_limited", "message": "Rate limit exceeded. Please try again later."}
       ```
       Cause: SerpApi rate limit hit. Fix: Wait and retry, or upgrade SerpApi plan.
     - **invalid_api_key** (from search.go, HTTP 401):
       ```json
       {"error": "invalid_api_key", "message": "Invalid SerpApi API key. Check your API key in the path or Authorization header."}
       ```
       Cause: Invalid or revoked SerpApi key. Fix: Verify key at https://serpapi.com/dashboard.
     - **forbidden** (from search.go, HTTP 403):
       ```json
       {"error": "forbidden", "message": "SerpApi API key forbidden. Verify your subscription and key validity."}
       ```
       Cause: Key doesn't have required subscription. Fix: Check SerpApi subscription status.
     - **search_error** (from search.go, catch-all):
       ```json
       {"error": "search_error", "message": "<context-specific error details>"}
       ```
       Cause: Various (invalid URL, HTTP error, JSON parse failure). Fix: Check error message for specifics.

7. **Troubleshooting** (USE-06, D-11/D-12)
   - **Symptom → Cause → Fix table** (D-11):
     | Symptom | Likely Cause | Fix |
     |---------|-------------|-----|
     | "Connection refused" | Wrong host/port | Check MCP_HOST and MCP_PORT; verify server is running |
     | "401 Unauthorized" or "Missing API key" | No API key provided | Include key in URL path or Authorization header |
     | "Invalid SerpApi API key" | Wrong or revoked key | Verify key at serpapi.com/dashboard |
     | "Engine not found" | Typo in engine name | Check serpapi://engines for valid names |
     | "Missing params" | Required parameter omitted | Check serpapi://engines/<engine> for required params |
     | "Rate limit exceeded" | Too many requests | Wait and retry, or upgrade SerpApi plan |
     | "Failed to load engine schemas" | Missing engines/ directory | Check --engines-dir flag; run from repo root or set ENGINES_DIR |
     | Empty search results | Engine requires additional params | Some engines need location, gl, hl params |
     | CORS errors in browser | Origin not allowed | Set --cors-origins to your client origin |
   - **Debug tips** (D-12):
     - Verbose logging: Server uses structured logging (slog) — check stderr output
     - Check engine schemas: Read `serpapi://engines` resource to list all engines, `serpapi://engines/<engine>` for params
     - Test with curl:
       ```bash
       # Health check
       curl http://localhost:8000/health
       # Search with path auth
       curl -X POST "http://localhost:8000/YOUR_KEY/mcp" -H "Content-Type: application/json" -d '{"jsonrpc":"2.0","method":"tools/call","params":{"name":"search","arguments":{"params":{"q":"test"}}},"id":1}'
       # Search with header auth
       curl -X POST "http://localhost:8000/mcp" -H "Authorization: Bearer YOUR_KEY" -H "Content-Type: application/json" -d '{"jsonrpc":"2.0","method":"tools/call","params":{"name":"search","arguments":{"params":{"q":"test"}}},"id":1}'
       ```
     - Correlation IDs: Each request gets a 32-char hex correlation ID; provide your own via `X-Correlation-ID` header for tracing
