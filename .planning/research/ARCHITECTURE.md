# Architecture Patterns

**Domain:** MCP server — search API gateway
**Researched:** 2026-04-15

## Recommended Architecture

```
┌──────────────────────────────────────────────────────────────────┐
│                         MCP Client                               │
│                  (Claude Desktop, Cursor, etc.)                   │
└──────────────────────────┬───────────────────────────────────────┘
                           │ Streamable HTTP (JSON-RPC)
                           │ /{API_KEY}/mcp
                           ▼
┌──────────────────────────────────────────────────────────────────┐
│                      Starlette + uvicorn                         │
│  ┌────────────────────────────────────────────────────────────┐  │
│  │                  Middleware Stack                          │  │

│  │                                                            │  │

│  │                                                            │  │

                                   ▼
```

### Component Boundaries

| Component | Responsibility | Communicates With |
|-----------|---------------|-------------------|
| CorrelationIdMiddleware (new) | Assign unique request ID, propagate to logs | All downstream logs and error responses |
| ApiKeyMiddleware | Extract & validate API key from path/header, store in request state | Starlette request pipeline |
| RequestMetricsMiddleware | Emit CloudWatch EMF metrics for each request | CloudWatch (via stdout logs) |
| FastMCP Instance | Register tools, resources, prompts; handle MCP protocol | MCP clients via JSON-RPC |
| Search Tool | Execute SerpApi searches with param validation & error handling | serpapi Python package, request.state for API key |
| Engine Schema Loader | Load, validate, and serve engine JSON schemas as MCP resources | engines/ directory |
| Healthcheck Handler | Return server health status | Load balancers, monitoring |
| Error Handler (planned) | Convert exceptions to structured MCP error responses | FastMCP ErrorHandlingMiddleware |

### Data Flow

```
Client → HTTP Request → CorrelationId MW → Metrics MW → Auth MW → CORS MW
  → FastMCP Router
    → If /healthcheck: return health JSON
    → If MCP request:
      → Parse JSON-RPC
      → Route to tool/resource/prompt handler
      → For search tool:
        → Validate engine name against known engines
        → Validate required params per engine
        → Inject API key from request state
        → Call serpapi.search()
        → Apply mode filter
        → Return JSON result / structured MCP error
      → Emit metrics (includes correlation ID)
      → Return response
```

## Patterns to Follow

### Pattern 1: In-Memory Testing with FastMCP Client
**What:** Test MCP tools without starting an HTTP server
**When:** All unit and functional tests
**Example:**
```python
from fastmcp import FastMCP, Client

async def test_search_tool():
    async with Client(mcp) as client:
        result = await client.call_tool("search", {"params": {"q": "test"}})
        assert result.data is not None
```

### Pattern 2: FastMCP Context for Observability
**What:** Use Context object for client-visible logging and progress reporting
**When:** Inside tool implementations
**Example:**
```python
from fastmcp import Context
from fastmcp.dependencies import CurrentContext

@mcp.tool()
async def search(params: dict[str, Any] = {}, mode: str = "complete",
                 ctx: Context = CurrentContext()) -> str:
    await ctx.info(f"Searching: engine={params.get('engine', 'google_light')}")
    await ctx.report_progress(progress=0, total=1)
    result = serpapi.search(search_params).as_dict()
    await ctx.report_progress(progress=1, total=1)
    return json.dumps(result)
```

### Pattern 3: Proper Exception Hierarchy for Error Handling
**What:** Catch specific exception types instead of string-matching error messages
**When:** Error handling in the search tool
**Example:**
```python
import serpapi

try:
    data = serpapi.search(search_params).as_dict()
except serpapi.exceptions.HTTPError as e:
    if hasattr(e, 'response') and e.response is not None:
        status = e.response.status_code
        if status == 429:
            return "Error: Rate limit exceeded..."
        elif status == 401:
            return "Error: Invalid API key..."
    return f"Error: {extract_error_response(e)}"
```

### Pattern 4: Engine Schema Validation at Startup
**What:** Validate all engine JSON files during lifespan setup, fail fast on errors
**When:** Server startup, before registering resources
**Example:**
```python
from fastmcp import FastMCP

@mcp.lifespan
async def engine_lifespan(server: FastMCP):
    engine_data = load_and_validate_engines()
    register_engine_resources(server, engine_data)
    yield {"engines": engine_data}
```

### Pattern 5: Request Correlation IDs
**What:** Assign unique IDs per request for cross-log tracing
**When:** As early in middleware stack as possible
**Example:**
```python
import uuid

class CorrelationIdMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request, call_next):
        request.state.correlation_id = (
            request.headers.get("X-Request-ID") or str(uuid.uuid4())
        )
        response = await call_next(request)
        response.headers["X-Request-ID"] = request.state.correlation_id
        return response
```

## Anti-Patterns to Avoid

### Anti-Pattern 1: String-Matching Exception Messages
**What:** `if "429" in str(e)` to check HTTP status codes
**Why bad:** Fragile, can match wrong exceptions, breaks if error message format changes
**Instead:** Access `e.response.status_code` directly, or catch specific exception subclasses. Verify SerpApi Python client exception properties.

### Anti-Pattern 2: Starting HTTP Server for Tests
**What:** Running `uvicorn` in test fixtures and making real HTTP requests
**Why bad:** Slow, flaky, requires port management, can't test edge cases easily
**Instead:** Use FastMCP's `Client` class for in-memory testing — no HTTP needed.

### Anti-Pattern 3: Untyped Dict Parameters Without Validation
**What:** `params: dict[str, Any] = {}` with no validation before forwarding to SerpApi
**Why bad:** Clients can send anything, errors are opaque, no discoverability
**Instead:** Validate params against engine schemas before calling SerpApi; return structured error messages with actionable guidance.

### Anti-Pattern 4: Dual-Channel Logging Without Correlation
**What:** CloudWatch EMF metrics AND Python logging with no way to correlate between them
**Why bad:** Can't trace a specific request across all log lines
**Instead:** Add request correlation IDs at middleware level; include in all log entries and CloudWatch dimensions.

### Anti-Pattern 5: Module-Level Side Effects for Resource Registration
**What:** Registering engine resources at module import time (`for _engine_path in _get_engine_files()`)
**Why bad:** Can't control initialization order, can't test without loading all engine files, can't validate before serving, server silently starts with broken engines
**Instead:** Use FastMCP's lifespan API for startup hooks. Load engines, validate, and register resources inside the lifespan context manager.

## Scalability Considerations

| Concern | At 100 users | At 10K users | At 1M users |
|---------|--------------|--------------|-------------|
| SerpApi rate limits | Per-key limits handle it | Per-key limits handle it | Per-key limits handle it (users upgrade plans) |
| Server throughput | Single uvicorn worker fine | May need multiple workers | Horizontal scaling with Copilot (already supported) |
| Memory (engine schemas) | ~100 JSON files, negligible | Same — schemas are static | Same — no per-request state |
| Cold start | <2s (Python + load 100 JSON files) | Same | Same — consider pre-warming |
| Connection limits | Default uvicorn settings fine | May need tuning | Use connection pooling to SerpApi |
